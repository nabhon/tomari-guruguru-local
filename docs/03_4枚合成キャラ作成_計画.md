# Plan: 4-sheet drift-free character creation (auto-composite)

## Context

When users generate the 6 expression sheets (A–F) with AI, each sheet is an independent generation, so colors drift slightly and stray spots appear between sheets. At runtime the app stacks the 6 sheets and swaps the active one, so the drift shows as a visible "pop" when the eyes blink or the mouth moves.

The reliable manual fix (documented in `docs/01_…txt`) is to composite every variant from one master so the body/hair/face is pixel-identical everywhere and only the eye/mouth region changes. This plan **automates that** and reduces the user's work from 6 generated sheets to **4**.

User decisions: locate/blend regions with **per-row band masks** (simplest, predictable); **add** this as a new option in the Add-character popup, keeping the existing 6-sheet flow.

## Inputs → outputs

**4 input sheets** (each a 5×5 angle sheet, same as today):
- `base` — eyes-open / mouth-closed (the master)
- `eyesClosed` — eyes-closed / mouth-closed
- `mouthHalf` — eyes-open / mouth-half
- `mouthOpen` — eyes-open / mouth-open

**Output**: the same 150 frames (A–F × 25 cells) the app already expects, so `writeCharacter`, slicing layout, runtime rendering, and `verify:pages` are all unchanged.

**Per-sheet composition recipe** (body always from `base` → zero drift):
| Sheet | = base + eye patch + mouth patch |
|---|---|
| A | base (no patch) |
| B | base + mouthHalf(mouth-band) |
| C | base + mouthOpen(mouth-band) |
| D | base + eyesClosed(eye-band) |
| E | base + eyesClosed(eye-band) + mouthHalf(mouth-band) |
| F | base + eyesClosed(eye-band) + mouthOpen(mouth-band) |

## How compositing works (per cell, on canvas)

For each cell `(r,c)`:
1. Draw the `base` cell as the full frame.
2. If the target sheet needs closed eyes, draw the `eyesClosed` cell into a temp canvas, mask it to the **eye band for row r** (feathered), then composite over the base.
3. If the target sheet needs a non-closed mouth, do the same with `mouthHalf`/`mouthOpen` and the **mouth band for row r**.
4. `canvas.toBlob(..., 'image/webp', 1)` → push `{ path: \`${sheet}/r${r}c${c}.webp\`, data }` (same shape as `sliceSheets`).

Patches are taken from the **same cell** of each variant sheet, so a profile cell is patched from the variant's profile cell — angle stays consistent; only pixel registration within the cell can vary (see risks).

### Per-row band masks
- `EYE_BAND[r] = {top, bottom}` and `MOUTH_BAND[r] = {top, bottom}` as fractions of cell height — one entry per row. Rows encode pitch (row 0 = strongly up → bands higher; row 4 = strongly down → bands lower), so the table is principled, not arbitrary.
- Horizontal extent: full width (optionally slight inset) with a feathered alpha gradient at all band edges so seams disappear.
- Feather via a temp-canvas alpha gradient + `globalCompositeOperation = 'destination-in'`.
- Defaults hardcoded as tuned constants in a new `src/sheet-composite.js`; easy to expose as tweaks later. Add a tiny "preview first cell" so the user can eyeball mask placement before committing all 150.

## Files

**New: `src/sheet-composite.js`**
- `EYE_BAND`, `MOUTH_BAND` (per-row fraction tables) + `FEATHER` default.
- `bandMask(ctx, cw, ch, band, feather)` — paint feathered alpha for a band.
- `compositeCell(cellCanvas, baseBmp, patches, r, c, cw, ch, masks)` — base + feathered patches.
- `sliceSheetsComposite(fileMap4, onProgress, masks?)` — mirrors `sliceSheets` (`talk-app.jsx:78`) but takes the 4 named inputs and emits all 150 A–F frames via the recipe table. Returns the same `[{path,data}]` array.

**Modify: `src/talk-app.jsx`**
- Add state: `addMode` (`'six' | 'four'`, default `'four'`), and a 4-slot file map (keys `base/eyesClosed/mouthHalf/mouthOpen`).
- Add-character popup (~line 360–380): a mode toggle (`TweakRadio`-style or two buttons) at the top. In 4-sheet mode show 4 labelled slots with descriptions; in 6-sheet mode show the existing A–F slots untouched.
- `doCreate` (~line 176): branch on `addMode` → `sliceSheetsComposite(files4, onProgress)` vs existing `sliceSheets(addFiles, onProgress)`. Validation requires all 4 (or all 6). Everything after (`writeCharacter`, refresh, select) is shared.
- Keep `SHEET_KEYS`/`SHEET_DESC` for the 6-sheet path; add a `FOUR_KEYS`/`FOUR_DESC` for the new one.

**Unchanged:** `writeCharacter`, `electron/characters.cjs createCharacter`, the dev `POST /__characters/create` middleware, runtime rendering, `verify:pages` (still A–F × 25 webp).

## Verification
- **Dev (`npm run dev`):** open "+ Add character" → 4-sheet mode → load the 4 sheets → preview first cell shows base with feathered eye/mouth bands aligned → Create → app slices 150 and selects the new char. Toggle blink and talk: the body must not change color at all between A↔D and mouth states (the whole point); only eyes/mouth swap. Switch to 6-sheet mode → original flow still works.
- **Electron (`npm run electron:dev`):** same, confirm `createCharacter` IPC writes the composited frames and the new character persists/reloads.
- **Edge cells:** check profiles (c0/c4) and extreme pitch (r0/r4) — verify the bands still cover eyes/mouth; tune `EYE_BAND`/`MOUTH_BAND` rows if a patch clips or misses.

## Risks / caveats
- **Pixel registration (main risk):** band masks assume the variant heads sit where the master's do. Since the user chose band-masks (no auto-align), small misalignment between independent generations can offset a patch. Mitigations: the hardened generation prompt (consistent centered heads + safe margins) reduces this; feathering hides small seams; advise generating the 4 sheets **in one session** for consistency. If misalignment is bad, the fallback upgrade is per-cell cross-correlation auto-align (deferred — explicitly out of scope per user choice).
- **Skin tone around the patch:** the eye/mouth patch carries the variant's local pixels; if a variant's skin differs slightly, the feather blends it. Usually invisible; widen feather if not.
- **Band tuning:** defaults are a starting point; the per-row tables likely need one tuning pass against a real character. The "preview first cell" makes this quick.
- **Inpaint alternative (note, not built):** if the user generates the 3 variants by inpainting the master (masking only eyes/mouth), inputs are pixel-aligned and drift-free by construction — compositing then becomes exact. Worth mentioning in the in-app guide.

## Out of scope
- Per-cell auto-alignment (cross-correlation) — deferred; user chose band-masks only.
- Replacing the 6-sheet flow — user chose to keep it as an option.
